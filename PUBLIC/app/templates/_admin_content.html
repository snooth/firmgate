Backup & recovery

SQLite database, portal branding, and uploads except Documents file storage. Role administrators only.

Backup

Downloads a zip containing the SQLite database (users, roles, groups, CRM, settings, Documents metadata, and more), files under instance/branding/ (portal logo), and everything under uploads/ except uploads/blobs/ (the content-addressed bytes for Documents, OnlyOffice, and similar). Export or copy uploads/blobs/ separately if you need those files. The filename includes the date and time in Australian eastern time (Sydney) when you start the download.

Store this file somewhere safe. Anyone with the zip can restore the entire portal, including user passwords as stored in the database.

Recovery

Upload a backup zip produced by this page. The current database is replaced; uploads/ is replaced from the archive except that uploads/blobs/ is left unchanged when the backup omitted it (new-format backups), so existing document files stay on disk. Older full backups that still include uploads/blobs/ restore those files as before. When the zip includes instance/branding/, portal logo files are replaced from the archive. Use only during a maintenance window; avoid using the app while a restore runs.

Development purposes

Populate demo content across the portal for testing and screenshots. Each click adds about 20% of the sample pool for every module (workforce, blogs, events, wiki, CRM, resource pool, security clearance, team chat, documents, games, home, and about company). Demo employee and contractor accounts use password demo123 and have MFA disabled. Progress is tracked so repeated clicks continue filling each section until complete. Cleared by factory reset.

Factory reset

Permanently wipes the SQLite database, all settings, uploads (including document file storage under uploads/blobs/), portal branding, CRM leads and pipeline data, and every other portal record. The application is restored to a fresh install with only the factory bootstrap administrator (admin@example.com / admin). CRM dashboard and pipeline views return to empty placeholders, and About Company “At a glance” stats are cleared. Download a backup first if you might need any of this data again. This cannot be undone.

Upgrade from Git

Safe in-place update: fetches and resets code to match the remote branch, keeps instance/ (database and document uploads), .env, and .venv — uploads are not copied during upgrade. Backs up only .env and the database file, reinstalls Python dependencies, verifies upload counts, then restarts the app service when configured. Requires a writable Git clone under the deploy path.

Upgrade from package

Upload a release ZIP when Git is unavailable. The package must include the full application tree, requirements.txt, and manifest.json (build with scripts/build_release_package.sh). The server keeps instance/, .env, and .venv, runs pip install -r requirements.txt, verifies uploads, then restarts the app service when configured. A light backup of .env and the database is taken first.
